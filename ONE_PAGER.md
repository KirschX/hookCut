# HookCut
## 컨텐츠를 유저가 클릭하고 싶어지는 Hook 콘텐츠로 바꾸는 AI Growth Tool

---

## 1. 왜 이 도메인을 선택했는가

이번 과제에서 AI 이미지 생성 도메인을 선택한 이유는, **텍스트 기반 컨텐츠의 매력을 짧고 시각적으로 매력적인 형태로 압축해 유입을 만드는 것**에 평소 관심이 있기도 했고, 뤼튼의 실제 서비스에도 적용해 임팩트를 만들어 볼 수 있다고 생각했기 때문입니다.

최근 AI 서비스는 단순 질의응답을 넘어, 캐릭터·스토리·상황극·팬덤형 경험으로 확장되고 있습니다. 이런 서비스에서는 좋은 캐릭터와 세계관을 만드는 것만큼이나, **유저가 왜 이 캐릭터를 눌러봐야 하는지, 왜 이 스토리에 들어가봐야 하는지**를 첫 접점에서 전달하는 것이 중요합니다.

저는 이 지점이 뤼튼의 AI 캐릭터/스토리형 서비스와도 맞닿아 있다고 보았습니다. 특히 캐릭터 기반 서비스에서는 대화나 스토리의 일부 장면, 감정선, 관계성, 반전 요소처럼 **짧게 잘라 보여줄 때 더 강하게 소비되는 훅 포인트**가 존재합니다. 이런 훅 포인트를 빠르게 추출하고, 이미지·컷 콘텐츠·광고 소재로 변환하는 문제는 충분히 제품화할 가치가 있다고 판단했습니다.

또한 최근 콘텐츠 소비가 이미지뿐 아니라 카드형, 만화형, 숏폼형 포맷으로 빠르게 확장되고 있다는 점도 고려했습니다. 이번 MVP는 컷형 이미지 중심으로 구현했지만, 추후 **광고 / 만화 / 숏폼형 하이라이트 콘텐츠로 확장 가능한 문제**로 보고 접근했습니다.

---

## 2. 어떤 타겟의 어떤 문제를 풀고자 했는가

HookCut의 초기 타겟은 **캐릭터·세계관·스토리형 콘텐츠를 운영하지만, 이를 짧은 Hook 콘텐츠로 전환하는 데 어려움을 겪는 개인 창작자와 소규모 운영자**입니다.

이들이 겪는 문제는 단순히 “이미지를 생성하기 어렵다”가 아닙니다.

### 핵심 문제

1. **긴 콘텐츠를 어떻게 짧게 잘라 보여줄지 어렵습니다**  
   캐릭터 설정, 긴 스토리, 대화 로그, 세계관 설명은 많지만, 어떤 장면이 유저의 클릭을 유도할지는 바로 판단하기 어렵습니다.

2. **범용 생성기는 제작은 도와주지만, 기획은 도와주지 않습니다**  
   일반 이미지 생성 서비스는 프롬프트를 넣으면 이미지를 만들어주지만, 어떤 장면을 강조할지, 어떤 감정선을 밀지, 어떤 타겟에게 어떤 톤으로 보여줄지는 사용자가 직접 정해야 합니다.

3. **콘텐츠를 외부 유입용 포맷으로 바꾸는 비용이 높습니다**  
   실제 유입은 긴 원문이 아니라 썸네일, 짧은 카피, 2컷/4컷 콘텐츠, 카드형 요약, 숏폼형 하이라이트에서 발생하는 경우가 많습니다. 하지만 이를 반복적으로 만드는 과정은 번거롭고 리소스가 듭니다.

4. **광고/홍보는 하나의 정답보다 여러 버전 실험이 중요합니다**  
   무엇이 먹힐지는 실제로 만들어봐야 알 수 있는데, 소규모 운영자는 여러 버전을 빠르게 실험하기 어렵습니다.

즉 사용자가 원하는 것은 “이미지 한 장”이 아니라,

> **캐릭터와 스토리의 매력을 유저가 클릭하고 싶어지는 Hook 콘텐츠로 빠르게 바꾸는 것**

입니다.

---

## 3. 기존 서비스들이 충분히 해결하지 못한 이유

기존 범용 이미지 생성 서비스는 대부분 다음 흐름에 머물러 있습니다.

> 프롬프트 입력 → 이미지 생성 → 마음에 안 들면 프롬프트 수정

이 방식은 사용자가 이미 “무엇을 보여줄지” 알고 있다는 전제를 가집니다. 그러나 실제 문제는 그 이전 단계에 있습니다.

- 어떤 장면을 뽑아야 할까요?
- 어떤 관계성과 감정선을 강조해야 할까요?
- 기존 팬이 아니라 신규 유저에게는 무엇을 먼저 보여줘야 할까요?
- 정적인 광고컷이 적합할까요, 카드형 요약이 적합할까요, 더 나아가 숏폼형 하이라이트가 적합할까요?

즉 기존 서비스가 해결하지 못하는 것은 생성 모델 자체보다,
**콘텐츠를 Hook 포맷으로 변환하는 판단과 워크플로우**입니다.

---

## 4. HookCut의 핵심 가설

HookCut은 다음 가설 위에서 설계했습니다.

> AI가 후킹 포인트를 먼저 추출하고 사용자는 선택만으로 방향을 정하는 가이드형 생성기는,
> 범용 free-prompt 생성기보다 사용자에게 더 매력적인 Hook 콘텐츠를 만든다.

여기서 Hook 콘텐츠는 단순 광고 배너만 의미하지 않습니다.
이번 MVP에서는 **컷형 이미지 중심**으로 다뤘지만, 본질적으로는 다음과 같은 포맷으로 확장 가능한 문제라고 보았습니다.

- 광고 배너
- 2컷 / 4컷 만화형 콘텐츠
- SNS 카드형 콘텐츠
- 캐릭터 소개 카드
- 스토리 하이라이트 카드
- 숏폼형 하이라이트 영상/영상 스토리보드

특히 AI 캐릭터/스토리 서비스에서는 유저 반응을 끌기 쉬운 장면, 대사, 관계성, 반전 포인트를 짧게 소비 가능한 형태로 재구성하는 니즈가 크다고 보았습니다. 이번 구현은 그 첫 단계로서 **컷형 이미지 Hook 콘텐츠**에 집중했습니다.

### 가설 검증 방법 (계획)

- 결과 화면에 좌우 비교 UI를 두고, HookCut 결과와 free-prompt 원샷 결과를 사용자가 직접 선택한다.
- free-prompt 측은 AI가 “범용 생성기에서 사용자가 쓸 법한 짧은 프롬프트”를 추정해 하나의 문장으로 만들고 그걸로 원샷 생성한다. 좌/우 위치는 매번 랜덤.
- `comparison_pick` 이벤트로 선택률을 적재.
- **판정**: HookCut 선택률이 50%(랜덤) 대비 유의하게 높으면 가설 검증.

이번 과제에서는 비교 UI 자체는 구현하지 않았습니다. 단계형 파이프라인 구조 위에 free-prompt 호출만 추가하면 붙도록 설계만 해두었습니다.

## 5. 어떻게 풀었는가

HookCut은 이미지 생성기라기보다, **캐릭터/스토리 기반 콘텐츠를 Hook 포맷으로 바꾸는 워크플로우**로 설계했습니다.

### 핵심 기능

#### 1) 콘텐츠 분석
사용자는 본문 발췌 텍스트, 매체(소설/웹툰/만화), 참고 URL(메타데이터), 추가 요청 메모를 입력합니다.  
AI는 이를 바탕으로 장르, 주요 인물, 핵심 갈등, 감정 톤, 훅 포인트, 스포일러 위험, 광고화 가능한 스토리라인 후보(3~5개)를 추출합니다.

#### 2) 스토리라인 / 훅 포인트 제안
HookCut은 바로 이미지를 생성하지 않습니다. 먼저 광고/홍보/하이라이트용으로 발전시킬 수 있는 방향을 제안합니다.

예시:
- 첫 만남의 임팩트
- 관계 역전 장면
- 감정선이 폭발하는 대사
- 세계관의 비밀이 드러나는 순간
- 유저 반응이 강할 만한 자극적 장면
- 숏폼형으로 잘릴 수 있는 하이라이트 상황

#### 3) 선택형 UI
사용자가 긴 프롬프트를 직접 작성하지 않아도 되도록, 핵심 질문을 선택형으로 설계했습니다.

예시:
- 연령대 / 독자 성향 / 기존 vs 신규
- 광고 목적 (첫 클릭 유도, 첫 화 읽기 유도, 캐릭터 어필 등)
- 톤앤매너 (강한 후킹, 감성 몰입, 어두운 분위기 등)
- 피하고 싶은 방향 (스포일러, 과도한 선정성, 캐릭터 왜곡 등)
- 광고 형식 (단일 배너, 2컷, 4컷 만화형, SNS 카드뉴스 등)
- 생성 전략 (안정 / 안정+실험 / 실험 위주)

#### 4) 다단계 생성 파이프라인
최종 결과는 한 번의 호출이 아니라, 단계별 API로 분리해 만들었습니다.

> 콘텐츠 분석(장르·인물·갈등·훅·스토리라인 후보) → 타겟·형식 선택 → 광고안(카피·CTA·컷 구성) 생성 → 컷 이미지 생성

이 구조를 통해 긴 입력을 직접 프롬프트화하는 부담을 줄이고, 각 단계에서 사용자가 방향을 조정할 수 있도록 했습니다.

#### 5) 결과물 생성 및 재실험
현재 MVP는 컷형 이미지를 생성하고, 함께 다음 정보를 제공합니다.

- 컷별 이미지와 구성 의도
- 헤드라인 · 서브 카피
- CTA
- 광고 의도(intent)
- 카피 클립보드 복사 / 광고안 메타데이터 JSON 다운로드
- 같은 전략으로 재생성 / 다른 전략으로 다시 시도

이를 통해 결과물이 단순 이미지가 아니라 “실험 가능한 콘텐츠 시안”으로 보이도록 설계했습니다.

## 6. 이번 MVP에서 구현에 집중한 범위

이번 과제에서는 문제를 넓게 설명하기보다, 실제로 동작하는 경험을 만드는 데 집중했습니다.

### 구현 범위
- 콘텐츠 입력
- AI 기반 콘텐츠 분석
- 훅 포인트 / 스토리라인 후보 생성
- 선택형 질문 UI
- 타겟 / 톤 / 형식 선택
- 단계형 생성 파이프라인
- 컷형 이미지 결과 생성
- 결과 확인 및 재생성 흐름

### 의도적으로 제외한 범위
- 고급 이미지 편집기
- 광고 플랫폼 직접 연동
- 숏폼 영상 자동 렌더링
- 대규모 협업 기능
- 정교한 성과 분석 대시보드

즉 이번 구현은 “모든 포맷을 다 지원하는 완성형 도구”보다,
**Hook 콘텐츠 생성 워크플로우가 실제로 더 나은 사용자 경험이 되는지**를 검증하는 데 초점을 맞췄습니다.

---

## 7. 왜 이 접근이 확장 가능하다고 보는가

이번 MVP는 이미지 도메인 안에서 시작했지만, 제가 본 핵심 문제는 더 넓습니다.

> 긴 캐릭터/스토리형 콘텐츠를, 외부 유입과 클릭을 만드는 짧은 Hook 콘텐츠로 변환하는 문제

이 문제는 정적인 컷 이미지에서 끝나지 않습니다. 같은 분석 구조를 활용하면 다음과 같이 확장할 수 있습니다.

- 캐릭터 홍보용 카드
- SNS용 컷 만화
- 광고용 배너
- 대화 하이라이트 카드
- 숏폼형 스토리보드
- 숏폼 영상용 장면/자막 초안

같은 방향으로 자연스럽게 확장될 수 있습니다.

특히 최근에는 콘텐츠 소비가 이미지 → 카드 → 숏폼 영상으로 빠르게 이동하고 있기 때문에, 캐릭터/스토리의 하이라이트를 포맷에 맞게 변환하는 문제는 앞으로 더 중요해질 가능성이 높다고 보았습니다.

따라서 HookCut은 현재는 컷형 이미지 기반 MVP이지만,
장기적으로는 **AI 캐릭터/스토리형 콘텐츠의 발견과 확산을 돕는 멀티포맷 Growth Layer**로 발전할 여지가 있다고 판단했습니다.

## 8. 성공을 어떻게 판단할 것인가

이번 MVP의 핵심 성공 기준은 “이미지가 생성되었는가”가 아닙니다.

더 중요한 질문은 다음입니다.

> 사용자가 이 결과물을 보고,
> “이 정도면 캐릭터/스토리를 소개하거나 홍보하는 데 써볼 수 있겠다”고 느끼는가입니다.

### 핵심 지표

**HookCut vs free-prompt 결과 선택률**

> 결과 화면 좌우 비교 UI에서, 사용자가 HookCut 결과를 선택한 비율 (4번 섹션 참조)

HookCut의 목적은 단순 이미지 생성이 아니라, 캐릭터와 스토리의 매력을 실제 공유·홍보에 활용할 만한 Hook 콘텐츠로 바꾸는 것이기 때문에, 같은 원본을 범용 free-prompt 생성기로 만든 결과보다 사용자가 직접 더 매력적이라고 평가하는지를 가장 중요한 신호로 봅니다.

**보조 신호**로는 사용자가 결과물을 “꺼내가는” 행위(카피 복사 / 광고안 데이터 다운로드)도 함께 추적할 수 있습니다.

### 보조 지표

- **Activation**: 콘텐츠 입력 완료율, 분석 결과 확인 후 다음 단계 이동률, 스토리라인 선택률, 결과 생성 완료율
- **Value**: 카피 복사율, 광고안 JSON 다운로드율, 결과 페이지 도달 후 컷 확대(zoom) 인터랙션 비율
- **Iteration**: 같은 전략으로 재생성한 비율, 다른 전략(타겟·형식·생성 전략)으로 돌아가 재시도한 비율

재생성률은 단독으로 해석하지 않습니다. 복사/다운로드 없이 반복 재생성된다면 결과 품질 불만족일 수 있고, 복사/다운로드 이후 추가 생성된다면 더 다양한 시안을 실험하려는 긍정 신호일 수 있어, 두 패턴을 분리해 해석합니다.

## 9. 정리

HookCut은 범용 이미지 생성기와 다르게,
**캐릭터와 스토리의 매력을 짧고 강한 Hook 콘텐츠로 바꾸는 경험**에 집중한 제품입니다.

이번 MVP에서는 컷형 이미지 중심으로 구현했지만, 문제 정의 자체는 더 넓게 보았습니다.

- 좋은 캐릭터와 스토리를 만듭니다
- 그중 유저가 반응할 만한 하이라이트를 찾습니다
- 이를 클릭과 유입을 만드는 짧은 포맷으로 압축합니다
- 여러 버전으로 빠르게 실험합니다

이 흐름은 광고 이미지뿐 아니라 만화형, 카드형, 나아가 숏폼형 콘텐츠로도 확장 가능합니다.

결국 HookCut이 풀고자 한 문제는 “이미지 생성” 그 자체보다,

> **AI 캐릭터/스토리형 콘텐츠를 어떻게 더 발견되기 쉽고, 클릭되기 쉽고, 공유되기 쉬운 형태로 바꿀 것인가**

에 가깝습니다.

이번 과제에서는 그 문제를 컷형 이미지 기반의 실제 동작하는 제품 경험으로 먼저 풀어보았습니다.
